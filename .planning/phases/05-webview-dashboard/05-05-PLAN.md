This is the critical integration task. Wire the dashboard into the extension lifecycle.

**In DashboardProvider.ts -- add data transformation:**

Add a static helper method `buildDashboardData` that transforms the extension's internal data (TimeBuckets, StatusBarData) into the webview's `DashboardData` format:

```typescript
public static buildDashboardData(
  buckets: TimeBuckets,
  statusBarData: StatusBarData,
  planType: string
): DashboardData
```

This method must:
1. Extract token breakdown from today's daily bucket (or sum of session buckets in 5hr window)
2. Convert rate limit info: Map RateLimitInfo -> RateLimitData (serialize Date to ISO string)
3. Compute session timing: windowStart from earliest session in 5hr window, windowExpiry = windowStart + 5h, timeRemainingMinutes
4. Build trend data: Convert daily/weekly/monthly Maps to sorted TrendDataPoint arrays
5. Compute session comparison: currentSessionTokens from 5hr window, averageSessionTokens from all sessions, sessionCount
6. Include metadata: lastUpdated, filesProcessed, linesSkipped, planType

For trend data period switching, handle `changePeriod` messages in `resolveWebviewView`:
- Store the current period preference in the provider
- When `changePeriod` received, rebuild trend data from the stored buckets for the new period
- Post `{ type: 'usageData', payload: updatedData }` with the new period's trend data

Add a method `updateBuckets(buckets: TimeBuckets, statusBarData: StatusBarData, planType: string)` that:
1. Stores the raw buckets (needed for period switching)
2. Calls `buildDashboardData()` to transform
3. Calls `updateData()` to push to webview
