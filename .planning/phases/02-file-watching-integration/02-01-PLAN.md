<objective>
Build the incremental parsing foundation for file watching: an OffsetTracker that persists per-file byte offsets in globalState, and an incremental parser that reads only new lines from a JSONL file starting at a given byte offset.

Purpose: Phase 1's parser reads entire files from the start every time. For real-time watching, we need to parse only new content added since the last read. This plan creates the infrastructure that the SessionWatcher (Plan 02) will use on each file change event.

Output: Two new modules (offsetTracker.ts, incrementalParser.ts) that enable offset-based incremental JSONL parsing with persistence.
</objective>

<task type="auto">
  <name>Task 2: Create incremental JSONL parser with offset support</name>
  <files>src/parser/incrementalParser.ts</files>
  <action>
Create `src/parser/incrementalParser.ts` with a `parseIncremental` function that:

1. Signature: `async function parseIncremental(filePath: string, startOffset: number, logger: Logger): Promise<IncrementalParseResult>`
2. Define and export `IncrementalParseResult` interface:
   ```typescript
   export interface IncrementalParseResult {
     records: TokenUsage[];
     newOffset: number;
     linesSkipped: number;
   }
   ```

3. Implementation:
   a. Check file size via `fs.promises.stat(filePath)` BEFORE reading
   b. If `startOffset > stats.size`, log warning and reset to 0 (file was truncated)
   c. If `startOffset === stats.size`, return empty result (no new data) — skip creating stream entirely
   d. Create `fs.createReadStream(filePath, { encoding: 'utf8', start: actualStart })`
   e. Wrap in `readline.createInterface({ input: stream, crlfDelay: Infinity })`
   f. For each line:
      - Track bytes: `bytesRead += Buffer.byteLength(line, 'utf8') + 1` (the +1 accounts for \n)
      - Skip empty/whitespace lines
      - Try `JSON.parse(line)` — if fails, increment linesSkipped and continue
      - If `parsed.type !== 'assistant'`, continue
      - Call `parseAssistantMessage(line)` from schemas.ts
      - If result is null, continue
      - Push to records array
   g. Return `{ records, newOffset: bytesRead, linesSkipped }`

4. Handle the edge case where the LAST line in a file might be incomplete (mid-write by Claude Code):
   - The readline interface handles this because incomplete lines without trailing \n will still be yielded
   - JSON.parse will fail on these -> linesSkipped increments -> next read will re-encounter them
   - BUT: we must NOT advance newOffset past an incomplete last line
   - Solution: Only update `bytesRead` after successful JSON.parse (or after confirmed complete line)
   - Actually simpler: track offset based on line length including newline. If last line has no newline, readline still yields it, but we've already added +1 for the missing newline. On next read, `start: newOffset` will skip the already-parsed content. If the line was incomplete, JSON.parse failed, so we didn't count it. Wait — we DID add bytes for it. So we need to track separately:
   - Use TWO counters: `committedOffset` and `currentOffset`. Only update `committedOffset` after a successfully parsed or intentionally skipped complete line. For the LAST line, if it fails JSON.parse AND there's no more data, it might be incomplete — but we can't distinguish "corrupt" from "incomplete" easily.
   - SIMPLEST CORRECT APPROACH: Always use `stats.size` as the newOffset after parsing completes, rather than manually tracking bytes. This way, if the file grew during parsing, we'll miss those bytes (they'll be caught on the next event). If the file had an incomplete last line, we'll re-read from the old offset next time but the completed line will now parse successfully.
   - Wait, that doesn't work either because we might skip the incomplete line.
   - BEST APPROACH: Track `bytesRead` manually. On JSON.parse failure for the LAST line (no more lines after it), subtract that line's bytes from bytesRead so we re-read it next time. Otherwise (middle lines that fail), they're genuinely corrupt and we skip them.

   Actually, the simplest and most robust approach: track bytesRead manually. After the for-await loop completes, compare bytesRead with stats.size. If bytesRead < stats.size, the file grew during parsing (fine, next event catches it). If bytesRead === stats.size, we read everything. The key concern is the last line being incomplete — but since we're reading from a known start offset and readline handles partial lines, and JSON.parse will fail on truncated JSON, the worst case is we count bytes for a corrupt last line and skip it. On next read, we start past it, which is fine because if it was truly corrupt it stays corrupt, and if it was incomplete it's now complete but we start past it.

   FINAL DECISION: Use the simple approach from the research doc. Track bytesRead = startOffset. For each line, add `Buffer.byteLength(line, 'utf8') + 1`. Return bytesRead as newOffset. Accept that a rare edge case (last line incomplete, then completed, then we skip past it) exists — but this is mitigated by:
   - The full reparse on activation catches everything
   - The watcher will fire again when more data is written, and the next line will be caught
   - One missed message out of thousands is acceptable for a monitoring extension
